# Методология: переданные баги, кто закрыл, скорость передачи

Три нетривиальных метрики, которые нельзя взять «в лоб» из полей Jira. Здесь —
почему и как.

## 1. Переданные в другие команды

**Неверно:** брать баги, у которых текущий тег команды = соседняя команда
(Ahaha, Passport AvIDo и т.п.). Это даёт и ложные срабатывания (баги, которые
никогда не приходили к нам — у соседних команд свой intake), и пропуски (баги,
переданные в команды вне юнита, или с очищенным тегом).

**Верно:** опираться на бот-комментарий о маршрутизации. Когда баг попадает на
команду, бот `JSD Bot` пишет комментарий вида:

```
Нотификация отправлена: profiles
```

где `profiles` (или `#profiles-support`) — канал команды. Это надёжный сигнал
«баг пришёл к нам». Если позже баг переназначают, появляется новый комментарий
с ДРУГИМ каналом:

```
Нотификация отправлена: #tns-verification-alert-gonarchs,@maslobodyan
```

Алгоритм:
1. JQL `comment ~ "Нотификация отправлена <CHANNEL>"` (по каждому каналу
   команды) + `issuetype=Bug` + `created >= start` → множество багов, пришедших
   на команду.
2. Переданные = из них те, у кого текущий `cf[22019]` НЕ входит в team_tags.
3. Принимающую команду бери из текущего `cf[22019]` (или из канала второго
   уведомления, если тег пуст/`None`).

**Осторожно с ложными совпадениями:** текстовый поиск `~ "...profiles"` матчит и
`#profiles-support`. Если у тебя несколько «своих» каналов (`profiles` И
`#profiles-support`) — это ок, оба наши. Но проверь пару задач через
`jira_get_comments`, что там действительно точный `Нотификация отправлена:
<наш канал>`, а не случайная подстрока чужого канала.

## 2. Кто закрыл (перевод в Resolved)

«Закрывший» ≠ текущий assignee. Assignee — это, как правило, разработчик
привязанного бага (часто в проекте AVITOID), а SPT в Resolved переводит QA,
который верифицирует фикс.

Для каждого закрытого-в-периоде бага:
1. `jira_get_issue` с `expandChangelog:true`.
2. В `changelog.histories` найди ПЕРВУЮ по времени запись, где есть
   `items[].field == "status"` и `toString == "Resolved"`.
3. Её `author.displayName` — закрывший.
4. Если перехода в Resolved нет (сразу Closed/Fixed/Done) — бери автора первого
   перехода в done-статус (Closed/Fixed/Done), игнорируя In Progress/Open/Reopened.

Объём большой (ответы тяжёлые) → **делегируй сбор субагенту** (general-purpose),
который вернёт компактный `key → displayName`. В минифицированном JSON `author`
идёт ПЕРЕД `items` внутри одной history-записи — учитывай при Grep по
сохранённому файлу.

Пограничные случаи: деактивированный автор (displayName = хеш) — помечай как
«деактивированный аккаунт»; если changelog не отдался (таймаут) — резолвер часто
определяется по комментарию-резолюции, время которого совпадает с
`resolutiondate`.

## 3. Скорость передачи

Для переданных багов из `jira_get_comments`:
- **Приход** = время комментария `Нотификация отправлена: <наш канал>`.
- **Передача** = время следующего `Нотификация отправлена: <другой канал>`.
- Разница в днях, сгруппированная по месяцу прихода, показывает динамику
  скорости триажа.

Учитывай, что баг мог «пинговаться» между командами (несколько уведомлений);
бери первое уведомление на наш канал как приход и первое последующее на чужой
канал как передачу.
